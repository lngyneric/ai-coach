### 4.2 关键发现

| 发现 | 说明 |
|:----|:------|
| 🔴 **Cache Read 占比 87.5%** | 近 14 亿 cache 读取 Token 占总费用大头 |
| 🟢 **Cache 利用率高** | 大量缓存命中说明会话复用良好 |
| 🟡 **Pro 模型几乎独占** | 仅 1.7% 调用使用 Flash，可优化 |
| 🟢 **无 Cache Create 费用** | 所有缓存为自动续期未产生新建费用 |

## 六、优化建议

### 🔴 高优先级

| 建议 | 预期节省 | 操作 |
|:----|:-------:|:-----|
| **简单任务切换 Flash** | 40-50% | 分析/文档/代码审查等任务使用 deepseek-v4-flash |
| **减少 Cache 重建** | 20-30% | 避免频繁新建会话或切换上下文 |

### 🟡 中优先级

| 建议 | 预期节省 | 操作 |
|:----|:-------:|:-----|
| **精简系统提示词** | 5-10% | 减少每轮 system prompt 长度 |
| **复用长会话** | 10-15% | 持续使用同一会话，保持缓存前缀命中 |

### 🟢 低优先级

| 建议 | 预期节省 | 操作 |
|:----|:-------:|:-----|
| 平台 AI 对话限流 | 极小 | 当前调用量 7 次/天，无需干预 |
| TTS 缓存 | 极小 | 已做过缓存优化 |

### 7.2 计算方法

```
每次API调用费用 = 
  (input_tokens / 1,000,000 × input_price) +
  (output_tokens / 1,000,000 × output_price) +
  (cache_read_input_tokens / 1,000,000 × cache_read_price) +
  (cache_creation_input_tokens / 1,000,000 × cache_create_price)
```
